# 体育馆羽毛球预约系统课程设计报告草稿

## 1 概述

### 1.1 选题描述

本系统面向高校体育馆羽毛球场地预约场景，解决传统人工登记、微信群预约中容易出现的信息不透明、重复预约、取消不及时和统计困难等问题。系统采用 A 类课程设计要求的 C/S 架构，普通用户通过桌面客户端完成注册、登录、查看场地、选择时间段、提交预约、取消预约和查看个人预约记录；管理员通过管理端维护用户、场地、时间段、系统公告和预约数据，并可导出预约统计结果。

### 1.2 目的和意义

通过本系统开发，可以综合运用 Python 面向对象编程、Tkinter 图形界面、SQLAlchemy ORM、MySQL 数据库、密码哈希、Excel 导出、单元测试和 PyInstaller 打包等技术，完成从需求分析、系统设计、编码实现、测试验证到发布交付的完整流程。

## 2 需求分析

### 2.1 系统概述

系统分为普通用户端和管理员端。客户端使用 Tkinter 实现图形化界面，服务端数据存储使用远程 MySQL，客户端通过 SQLAlchemy ORM 和 PyMySQL 驱动访问数据库。

### 2.2 功能需求

- 普通用户：注册、登录、查看场地、查看可预约时间段、提交预约、取消个人预约、查看个人预约记录。
- 管理员：用户启用/禁用、场地新增与停用、批量生成时间段、查看全部预约、取消异常预约、维护系统设置、导出预约数据。
- 系统设置：维护系统公告和单个用户每天最大预约次数。
- 数据导出：将预约记录导出为 Excel 文件。

### 2.3 非功能性需求

- 安全性：用户密码使用 Werkzeug 哈希保存，登录时按角色进入不同界面。
- 可维护性：使用 UI、Service、Model、Config 分层结构。
- 稳定性：SQLAlchemy 开启 `pool_pre_ping=True`，提升远程数据库连接可靠性。
- 可测试性：核心业务逻辑通过 pytest 覆盖。

## 3 系统设计

### 3.1 总体架构

系统采用 C/S 架构。客户端为 Python Tkinter 桌面程序，数据库服务器为远程 MySQL。客户端内部按界面层、业务服务层、ORM 模型层和配置层组织。

### 3.2 功能设计

- 登录注册模块：验证账号密码，普通用户注册默认 `role=user`。
- 角色控制模块：管理员账号 `role=admin`，登录后进入管理员窗口。
- 预约模块：校验时间段状态、每日预约次数限制，创建、取消或核销预约。
- 管理模块：维护用户、场地、时间段和系统设置。
- 统计导出模块：汇总预约数量并导出 Excel。

### 3.3 数据库设计

主要数据表包括 `users`、`courts`、`time_slots`、`reservations`、`settings`。推荐使用 `sql/init.sql` 初始化表结构和演示数据。

### 3.4 界面设计

登录窗口提供用户名、密码、手机号输入，以及登录和注册按钮。普通用户窗口包含场地查询、我要预约、我的预约三个标签页。管理员窗口包含用户管理、场地管理、时间段管理、预约管理、系统设置、统计导出六个标签页。

### 3.5 关键技术

- SQLAlchemy ORM：对象关系映射和事务管理。
- PyMySQL：MySQL 驱动。
- Werkzeug：密码哈希和校验。
- tkcalendar：日期选择控件。
- openpyxl：Excel 导出。
- PyInstaller：打包为 `.exe`。

## 4 系统实施

### 4.1 系统开发环境

- Python 3.9+
- macOS / Windows
- MySQL 8.x
- 虚拟环境 `.venv`

### 4.2 系统运行环境

运行前需要配置 `.env`，填写 `DB_HOST`、`DB_PORT`、`DB_USER`、`DB_PASSWORD`、`DB_NAME`。目标电脑需要可以访问远程 MySQL。

### 4.3 核心功能实现

代码入口为 `main.py`。数据库连接位于 `app/db.py`，ORM 模型位于 `app/models.py`，业务服务位于 `app/services/`，界面代码位于 `app/ui/`。

### 4.4 编码规范

模块名和函数名使用小写加下划线，类名使用大驼峰命名。界面层不直接操作数据库，统一调用服务层。异常以业务异常抛出，再由界面层转换为弹窗提示。
