# 数据库设计说明

## users

保存用户账号、手机号、密码哈希、角色和状态。`role` 字段用于区分管理员和普通用户。

约束：`role` 只允许 `user/admin`，`status` 只允许 `normal/disabled`。

## courts

保存羽毛球场地编号、名称、位置、开放状态和备注。

约束：`status` 只允许 `open/closed`。

## time_slots

保存某个场地在某天的开始时间、结束时间和可预约状态。对 `court_id + slot_date + start_time + end_time` 建唯一约束，避免重复生成时间段。

约束：`status` 只允许 `available/booked/disabled`，并要求 `start_time < end_time`。

## reservations

保存预约编号、用户、场地、时间段、预约状态、创建时间和取消时间。

预约创建时，服务层通过条件更新 `time_slots.status` 抢占时间段。只有当时间段仍为 `available` 时才会更新为 `booked` 并写入预约记录，以降低多客户端并发重复预约风险。

数据库层使用 `active_slot_id` 生成列：

```sql
CASE WHEN status = 'booked' THEN slot_id ELSE NULL END
```

并在 `active_slot_id` 上建立唯一约束 `uq_reservations_active_slot`。这样同一个时间段只能存在一条 `booked` 状态的有效预约，取消后的历史记录不会阻塞后续重新预约。

## settings

保存系统设置，例如 `max_daily_reservations` 和 `announcement`。

## 初始化脚本

`sql/init.sql` 是推荐的手工初始化入口，包含表结构、默认系统设置、演示场地和当天演示时间段。`schema.sql` 仅作为表结构参考。SQLAlchemy ORM 模型与 `sql/init.sql` 保持同一套关键约束。
